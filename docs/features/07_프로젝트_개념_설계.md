# 07. 프로젝트 개념 설계

> 미구현 로드맵입니다. 현재 모든 업무는 독립적(`project_id = null`)으로 존재합니다.

---

## 핵심 원칙

> `project_id = null` 이면 독립 스케줄로 처리 — 기존 업무/약속은 변경 없이 그대로 동작합니다.

---

## 현재 상태

```
현재 (구현됨)
  sm_tasks
    ├── id
    ├── title
    ├── task_date / due_date
    ├── task_time / duration
    ├── priority / category / status
    └── project_id = null  ← 모든 업무가 독립
```

---

## 설계: 추가될 데이터 구조

### sm_projects (신규 테이블)

```sql
CREATE TABLE sm_projects (
  id          uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  title       text NOT NULL,
  description text,
  color       text,
  start_date  date,
  end_date    date,
  created_by  uuid REFERENCES sm_users(id),
  created_at  timestamp DEFAULT now()
);
```

| 컬럼 | 타입 | 설명 |
|------|------|------|
| `id` | uuid | PK |
| `title` | text | 프로젝트명 |
| `description` | text | 프로젝트 설명 |
| `color` | text | 캘린더/칸반 표시 색상 |
| `start_date` | date | 프로젝트 시작일 |
| `end_date` | date | 프로젝트 종료일 |
| `created_by` | uuid | 생성자 (sm_users.id 참조) |

### sm_tasks 추가 컬럼

```sql
ALTER TABLE sm_tasks
  ADD COLUMN project_id      uuid REFERENCES sm_projects(id) ON DELETE SET NULL,
  ADD COLUMN parent_task_id  uuid REFERENCES sm_tasks(id) ON DELETE CASCADE,
  ADD COLUMN sort_order      integer DEFAULT 0;
```

| 컬럼 | 타입 | 설명 |
|------|------|------|
| `project_id` | uuid | 소속 프로젝트 (null = 독립 업무) |
| `parent_task_id` | uuid | 부모 업무 ID (null = 최상위 업무) |
| `sort_order` | integer | 동일 레벨 내 정렬 순서 |

---

## 독립 업무 vs 프로젝트 업무

```
project_id = null  →  독립 스케줄 (현재 동작 그대로)
project_id = UUID  →  프로젝트 소속 업무

parent_task_id = null  →  최상위 업무
parent_task_id = UUID  →  서브태스크 (하위 업무)
```

### 계층 구조 예시

```
프로젝트: "웹사이트 리뉴얼"
  └── 업무: "디자인 시안 작성"        (parent_task_id = null)
        ├── 서브태스크: "홈 페이지"   (parent_task_id = 디자인 시안 ID)
        └── 서브태스크: "마이페이지"  (parent_task_id = 디자인 시안 ID)
  └── 업무: "프론트엔드 개발"         (parent_task_id = null)
```

---

## 뷰별 동작 설계

| 뷰 | 동작 방식 | 비고 |
|----|----------|------|
| **캘린더** | 기존 TaskBar 재활용 (task_date ~ due_date) | 이미 구현됨 |
| **칸반** | 프로젝트 탭으로 필터링 | `project_id` 기준 분류 |
| **리스트 (신규)** | 계층형 트리 (펼침/접기) | `parent_task_id` 재귀 렌더링 |
| **간트 (신규)** | 주차 기준 타임라인 | `start_date ~ due_date` 바 표시 |

---

## 리스트 뷰 — 계층형 트리 렌더링

```jsx
// 재귀 컴포넌트 구조 (설계)
function TaskTreeItem({ task, depth = 0 }) {
  const children = tasks.filter(t => t.parent_task_id === task.id);

  return (
    <div style={{ paddingLeft: depth * 20 }}>
      <TaskRow task={task} />
      {children.map(child => (
        <TaskTreeItem key={child.id} task={child} depth={depth + 1} />
      ))}
    </div>
  );
}
```

---

## 간트 뷰 — 타임라인 설계

```
날짜축 (주 단위)
──────────────────────────────────
웹사이트 리뉴얼  [=========]
  디자인 시안    [====]
  프론트 개발         [=====]
```

- X축: 날짜 (주 단위 또는 일 단위)
- Y축: 프로젝트 > 업무 > 서브태스크 계층
- 바 너비: `due_date - task_date` 일수
- 기존 CalendarGrid의 `TaskBar` 컴포넌트 로직 재활용 가능

---

## 구현 우선순위 (로드맵)

| 단계 | 작업 | 예상 범위 |
|------|------|----------|
| 1단계 | `sm_projects` 테이블 생성 + 업무 생성 모달에 프로젝트 선택 UI 추가 | DB + 모달 |
| 2단계 | 리스트 뷰: `parent_task_id` 기반 계층형 트리 렌더링 | 신규 컴포넌트 |
| 3단계 | 간트 뷰: 주차 기준 타임라인 (TaskBar 로직 재활용) | 신규 페이지 |
| 4단계 | 칸반보드 프로젝트 탭 필터링 | 기존 컴포넌트 수정 |

---

## 마이그레이션 안전성

- 기존 `sm_tasks` 레코드는 `project_id = null` 상태 유지 → 캘린더/칸반 동작 변화 없음
- `parent_task_id = null` → 기존 업무는 모두 최상위 레벨로 간주
- 신규 컬럼은 모두 nullable이므로 기존 코드 수정 없이 점진적 도입 가능
